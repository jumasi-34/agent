# [Raw] TP OE 2026 Target Definition and Integration Decision

## 1. 발생 배경 및 맥락
TP 공장(PLANT = 'TP')의 신차 개발 프로젝트 생산 달성률과 품질 데이터를 모니터링하기 위한 시스템 구축 과정에서, 분석 대상 규격을 정의하는 비즈니스 룰 및 서로 다른 물리 데이터베이스 간 데이터 정합성을 확보하는 데 중대한 의사결정이 필요했습니다.

## 2. 발견된 문제 및 해결 과정
1. **타깃 연도 및 규격 조건의 모호성**:
   * 최초 기획 과정에서 MP-GATE 연도가 2025년인 규격을 조회하려 했으나, 2026년 이상(>= 2026) MP-GATE 통과 규격으로 조건이 긴급 정정되었습니다.
   * `staging.db`의 `iqm_plus_spec_master` 테이블 내부 `MP_GATE_DT` 칼럼은 `YYYY-MM-DD` 문자열(TEXT) 형식으로 적재되어 있었습니다.
   * 이에 따라 SQLite 환경에서 가장 우수한 쿼리 성능과 데이터 추출 정확도를 유지하기 위해 `SUBSTR(MP_GATE_DT, 1, 4) >= '2026'` 조건식을 수립했습니다.
   * 이 필터를 통해 `PLANT = 'TP'` 조건과 만족하는 최종 타깃 자재코드(M-Code) 5대(`1033649`, `1033647`, `2022188`, `1034831`, `1034828`)를 명확하게 도출 및 매핑했습니다.

2. **이종 DB 간 조인 데이터 불일치 이슈**:
   * `staging.db`의 `iqm_plus_spec_master` 내 `MCODE` 칼럼과 `ops.db`의 `product_audit_regular_development` 내 `M-code` 칼럼 간 데이터 조인을 수행해야 했습니다.
   * 분석 결과, 이 두 컬럼의 데이터 타입이 텍스트(TEXT)와 정수(INTEGER)로 불일치하여 단순 조인 시 누락 현상이 발생할 가능성이 있었습니다.
   * 이 정합성 문제를 예방하기 위해, 데이터 로딩 쿼리 단계에서 `CAST(MCODE AS INTEGER)` 처리를 명시적으로 수행하여 Pandas `pd.merge` 병합 시 완벽한 Outer-Join 데이터 정합성을 확보했습니다.

3. **SQLite 트랜잭션 롤백 및 세션 관리 정합성**:
   * `st.data_editor`를 이용한 M-Code 추가, 수정, 삭제 일괄 가공 시, 부분적인 무결성 실패가 데이터베이스 전체를 오염시킬 위험이 있었습니다.
   * `conn.isolation_level = None`과 `BEGIN TRANSACTION / COMMIT / ROLLBACK` 명시적 DML 설계를 통해 일괄 트랜잭션 수동 커밋 제어 장치를 완비했습니다.
   * Streamlit `st.data_editor` 세션 저장소 버퍼가 갱신 후 잔존하는 문제를 예방하기 위해, 커밋 성공 시 `st.session_state`에서 에디터 위젯 세션 키를 `pop`하고 페이지를 `st.rerun()` 시키는 "세션 가로채기(Interception)" 패턴을 수립했습니다.

## 3. 학습 및 시사점
- SQLite 문자열 연도 필터링 시 `SUBSTR` 함수의 활용은 간단하지만 인덱스를 최적으로 태울 수 있는 비즈니스 모집합 도출에 유용합니다.
- 데이터 병합 시 칼럼의 명시적 Type Casting (`CAST(col AS TYPE)`)은 분산된 이종 DWH 데이터 적재 파이프라인에서 가장 먼저 챙겨야 할 데이터 가드레일입니다.
- Streamlit 위젯은 값을 직접 수정하려 하면 API 예외를 유발하므로, 인스턴스화 직전에 세션 상태를 인터셉트하여 청소(pop)하는 지혜가 필요합니다.
