# Architecture Guide (아키텍처 설계 및 계층 분리 가이드)

## 1. 왜 존재하는가 (Why)
이 가이드는 프로젝트의 뼈대를 이루는 3-Layer(UI-Service-Query) 물리 격리 사상을 명확히 정의하고, 각 계층이 월선하거나 오용되어 스파게티 코드로 변질되는 것을 원천 차단하기 위해 존재합니다. 엄격한 물리 격리는 코드의 결합도를 낮추고 테스트 용이성을 극대화하며, 인간 개발자와 에이전트 간의 병렬 개발 중 간섭을 완전히 제어하는 근간이 됩니다.

## 2. 어디와 연결되는가 (Connections)
- **UI 및 프레젠테이션 설계**: [[Streamlit UI Development]]를 통해 화면 및 위젯 제어와 완전히 연계됩니다.
- **시각화 데이터 흐름**: [[Plotly Visualization System]]에서 드로잉되는 차트 데이터의 원천 바인딩 및 격리 모델과 결합됩니다.
- **무결성 및 정적 컴파일 검증**: [[Harness Testing & Quality Gate]]를 통해 계층 분리 위배 여부를 실시간으로 정밀 진단합니다.

## 3. 무엇을 이해해야 하는가 (What)
- **3-Layer의 물리적 경계**:
  - **UI Layer (`app/` 하위 화면 페이지 및 공통 컴포넌트)**: 오직 위젯 렌더링, 최종 룩앤필 레이아웃, 사용자 반응 수집만을 처리합니다. 어떠한 경우에도 직접 SQL 쿼리를 기안하거나 가공하지 않습니다.
  - **Service Layer (데이터 변환 및 전처리 비즈니스 엔진)**: UI 계층이 요청한 데이터를 Pandas 데이터프레임 형식으로 받아 비즈니스 수식 및 포맷팅 처리를 전담합니다. 
  - **Query Layer (데이터베이스 접근 및 조회 격리)**: SQLite 데이터베이스에 직접 접근하는 유일한 관문으로, 표준 SQL을 사용하여 데이터를 추출하는 함수들로 구성됩니다.
- **디스플레이 명칭 분리 원칙**:
  - SQL 쿼리 내부에 한글 별칭을 하드코딩(`AS "한글"`)하는 것은 엄격히 금지됩니다. 모든 물리 쿼리는 영문 필드명을 보존해야 하며, 가공 및 한글 맵핑은 서비스 계층 및 메타데이터를 거쳐 UI 레이어로 전달되어야 합니다.
